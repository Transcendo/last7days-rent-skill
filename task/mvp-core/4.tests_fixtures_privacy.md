# 4. 测试、Fixtures 与隐私校验

## Agent 角色

QA/privacy subagent。可与 `4.integration_dry_run_docs.md` 并行执行。

## 目标

建立离线可验证测试体系，覆盖 profile、schema、source、隐私、去重、评分、渲染和 CLI smoke。核心目标是确保功能可校验通过，而不是只写文档或样子代码。

## 依赖

- 任务 2 完成。
- 任务 3 的功能模块尽量完成；若部分未完成，先写 failing/xfail 标注清楚。

## 必读上下文

- 所有已完成任务文件。
- `docs/product-contract.md`
- `docs/mvp-boundaries.md`
- `docs/architecture.md`

## 写入范围

只允许编辑：

- `tests/`
- `tests/fixtures/`
- `pyproject.toml` 中测试配置。

不要编辑业务实现，除非只是修正显然的测试导入路径。

## 实现要求

- 建立 fixtures：
  - profile onboarding。
  - 贝壳/链家列表 HTML。
  - Wellcee JSON-LD。
  - 房天下列表。
  - 非 MVP 来源样本，用于验证会被拒绝或标记 out of scope。
  - 敏感文本样本，含手机号、微信号、群名、姓名，仅用于隐私守卫测试，不代表 MVP 支持私域导入。
  - 重复房源。
  - 低价异常房源。
  - 缺字段房源。
- 建立测试：
  - `test_schema.py`
  - `test_profile.py`
  - `test_privacy.py`
  - `test_sources_*.py`
  - `test_dedupe.py`
  - `test_scoring.py`
  - `test_render.py`
  - `test_cli.py`
- 测试必须证明：
  - P1/P2/私域/WebSearch 来源不会进入 MVP listing pipeline。
  - L1 不会被写成已验真。
  - 缺失字段不被补全。
  - 报告不含手机号、微信号、群名、姓名。
  - fixture/offline search 能产出报告。

## 不做的事

- 不依赖真实网络才能通过测试。
- 不保存真实用户数据。
- 不测试验证码绕过。
- 不把大批量线上 HTML 塞进 repo。

## 验收标准

- `python -m pytest` 可运行。
- 隐私测试必须通过。
- CLI smoke 测试覆盖 `--help` 和 fixture search。
- 测试失败时能指出哪个契约未满足。
