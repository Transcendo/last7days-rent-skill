# 3. Dynamic Workflow Agents 文档

## Agent 角色

Workflow docs subagent。可与其他 `3.*` 任务并行执行；只处理多 agent 工作流文档。

## 目标

把 `last7days-rent-skill` 的开发和运行方式设计成 dynamic workflow agents：用户只看到一个入口 prompt，main orchestrator 负责启动多个 subagent 并合并结果，避免单 context window 爆掉。

重要原则：multi-agent 是为了准确、高效地完成任务，不是必须强加的流程。如果任务很小、强依赖同一上下文、写入范围高度重叠，或拆分会增加沟通和集成成本，main orchestrator 应选择单 agent 串行完成。

## 依赖

- 任务 1 完成。

## 必读上下文

- `docs/product-contract.md`
- `docs/mvp-boundaries.md`
- `docs/architecture.md`
- 本目录下全部 task 文件。

只允许编辑：

不要编辑 README，除非任务 `3.readme_install_prompts.md` 已完成并要求补链接。

## 实现要求

- 定义执行角色：
  - `main-orchestrator`
  - `profile-agent`
  - `source-agent`
  - `privacy-risk-agent`
  - `ranking-render-agent`
  - `docs-agent`
  - `qa-agent`
  - `integration-agent`
- 明确并行规则：
  - 同编号 task 可并行。
  - 同编号任务不得编辑同一写入范围。
  - schema/runtime 完成后，profile/source/ranking/docs 可并行。
  - final integration 必须最后做。
- 明确不适合 multi-agent 的情况：
  - 任务足够小，单 agent 可在当前 context 内准确完成。
  - 任务需要持续共享同一段复杂上下文。
  - 多个任务会频繁修改同一核心文件。
  - 拆分后的 handoff 成本高于执行收益。
  - 需要主 agent 做产品判断、隐私判断或最终验收。
- 明确交接物：
  - 每个 subagent 输出 changed files、tests run、warnings、remaining risks。
  - 主 agent 负责最终校验，不把 subagent 结果直接当验收。
- 提供可复制 Codex goal prompt：
  - 读取 `task/mvp-core/`。
  - 启动多个 subagent。
  - 每个 subagent 遵守写入范围。
  - 主 agent 最后跑 validator、pytest、CLI smoke、git status。
- 说明用户使用时的 dynamic workflow：
  - profile agent 建档。
  - source agents 查不同信源。
  - privacy/risk agent 脱敏和风险。
  - ranking/render agent 输出短名单。

## 不做的事

- 不实现真实 agent runtime。
- 不绑定某个私有 vendor API。
- 不让 subagent 处理未脱敏私域原文。

## 验收标准

- 文档清楚列出并行/串行任务表。
- 文档包含可直接复制的 goal mode prompt。
- 文档解释如何防止 context window 爆掉。
- 文档明确说明：不适合 multi-agent 时不要强行拆分，目标是准确和高效。
- 文档强调隐私和写入范围。
